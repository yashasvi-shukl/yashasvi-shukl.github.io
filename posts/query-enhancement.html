The problem: users rarely phrase their question the way the relevant document was written. Query expansion bridges that gap before retrieval even starts.

Why a Single Query Isn't Enough

You've built a great retrieval system. Your embeddings are good. Your vector store is fast. And yet, occasionally your RAG system misses obvious answers. The document is right there in the index, but the retriever didn't surface it.

The culprit is usually a vocabulary mismatch between the user's query and the document's language. A user asks: "What should I do if the landing gear fails to retract?". The relevant manual section is titled: "Non-standard retraction of the main undercarriage — troubleshooting procedure". Same concept. Different words. Without query expansion, the embedding distance between these might not be small enough to rank in the top-k.

Query expansion is the practice of generating alternative formulations of the user's query before (or alongside) retrieval, then merging the results.

Multi-Query Generation

The simplest and most effective form of query expansion: ask an LLM to rewrite the query from multiple perspectives.

MULTI_QUERY_PROMPT = """You are an expert at reformulating search queries to improve document retrieval.

Given a user question, generate {n} alternative versions that:
- Use different vocabulary and phrasing
- Approach the question from different angles
- Maintain the same core information need

Original question: {question}

Output ONLY the alternative questions, one per line. No numbering, no explanation."""

async def generate_multi_queries(question: str, n: int = 3) -> list[str]:
    response = await llm.ainvoke(
        MULTI_QUERY_PROMPT.format(question=question, n=n)
    )
    alternatives = [q.strip() for q in response.content.strip().split('\n') if q.strip()]
    return [question] + alternatives  # include original

# Example output for "What causes engine oil pressure to drop?"
# → "What are the reasons for low engine oil pressure?"
# → "Engine oil pressure warning light — possible causes and diagnosis"
# → "How to troubleshoot falling oil pressure in aircraft engines"

Each alternative query is then embedded and searched independently. Results from all queries are merged and deduplicated, usually via Reciprocal Rank Fusion.

Hypothetical Document Embeddings (HyDE)

HyDE flips the model: instead of expanding the query, you use the LLM to generate a hypothetical answer to the question, then embed that hypothetical answer and use it as the search vector.

The intuition is that a hypothetical answer — even if factually wrong — will be written in language closer to how actual documents are written, making it a better retrieval vector than the raw question.

HYDE_PROMPT = """Write a detailed, technical answer to the following question.
Your answer should be written in the same style as a technical manual or documentation.
It may not be factually correct — it just needs to be plausible and written in document-like language.

Question: {question}

Answer (2-3 paragraphs):"""

async def hyde_retrieve(question: str, vector_store, top_k: int = 10):
    # Generate hypothetical document
    hyp_response = await llm.ainvoke(HYDE_PROMPT.format(question=question))
    hypothetical_doc = hyp_response.content

    # Embed the hypothetical document
    hyp_embedding = embedding_model.embed_query(hypothetical_doc)

    # Search using the hypothetical embedding
    results = vector_store.similarity_search_by_vector(hyp_embedding, k=top_k)
    return results

HyDE is especially effective for technical domains where user questions are informal but documents are formal. The transformation from question-language to document-language is where a lot of retrieval failures hide.

Query Enhancement — Cleaning Before Expanding

Before expanding, it's worth cleaning. Real user queries in enterprise settings often contain:

  • Typos and misspellings
  • Domain abbreviations ("MEL" for Minimum Equipment List, "AOG" for Aircraft on Ground)
  • Grammatically incomplete fragments
  • Informal phrasing that's technically ambiguous
QUERY_ENHANCEMENT_PROMPT = """Clean and enhance the following search query for a technical documentation system.

1. Fix spelling errors
2. Expand known abbreviations: {abbreviations_json}
3. Make the query grammatically complete
4. Do not change the meaning or add new information

Original query: {query}

Return ONLY the enhanced query, nothing else."""

# Abbreviations were extracted once using LLM over all documents
# and stored in a JSON file
import json
with open("abbreviations.json") as f:
    abbrevs = json.load(f)

async def enhance_query(query: str) -> str:
    response = await llm.ainvoke(
        QUERY_ENHANCEMENT_PROMPT.format(
            query=query,
            abbreviations_json=json.dumps(abbrevs, indent=2)
        )
    )
    return response.content.strip()

Putting It Together — The Full Query Pipeline

async def full_query_pipeline(raw_query: str, vector_store) -> list:
    # Step 1: Clean and enhance
    enhanced_query = await enhance_query(raw_query)

    # Step 2: Generate alternative perspectives
    all_queries = await generate_multi_queries(enhanced_query, n=3)

    # Step 3: Retrieve for each query concurrently
    import asyncio
    retrieval_tasks = [
        vector_store.asimilarity_search(q, k=15)
        for q in all_queries
    ]
    all_results = await asyncio.gather(*retrieval_tasks)

    # Step 4: Merge via RRF
    ranked = reciprocal_rank_fusion(all_results)

    # Return top 20 unique documents
    return ranked[:20]

The concurrency is important. Running 4 queries sequentially adds 4× the retrieval latency. Running them concurrently costs almost the same as a single query.

Tradeoffs and When to Use Each Technique

TechniqueLatency CostBest ForAvoid When
Query cleaning~Low (+1 LLM call)Always worth it—
Multi-query (×3)Medium (+retrieval×3)Vocabulary mismatch likelyTight latency budgets
HyDEMedium (+1 LLM + retrieval)Informal → formal domainShort factual queries
Multi-query + HyDEHigherMaximum recallReal-time responses

In high-stakes RAG (where a missed document is costly), the latency overhead of multi-query + HyDE is almost always worth it. In conversational interfaces where users expect <1s responses, query cleaning alone gives you most of the benefit at a fraction of the cost.
