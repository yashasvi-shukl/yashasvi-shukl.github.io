Parent-Child Chunking: The RAG Pattern Nobody Talks About Enough

Standard chunking breaks context. Parent-child chunking preserves it — giving you retrieval precision and generation context at the same time.

RAG ~ 7 min read

Core idea: index small chunks for retrieval precision, but answer with their full parent context. Two stores, one query, dramatically better results.

The Problem with Standard Chunking

Every RAG tutorial follows the same recipe: split your document into chunks of ~512 tokens, embed them, push them into a vector store, retrieve top-k at query time. Clean. Simple. It works — until you're dealing with real documents that have real structure.

The fundamental tension in RAG chunking is this: small chunks win at retrieval (tight, focused semantic signal) but large chunks win at generation (the LLM needs enough context to actually answer). Standard fixed-size chunking forces you to pick a compromise that's suboptimal at both ends.

Split too small and you retrieve fragments — a sentence that's meaningless without the surrounding paragraph. Split too large and your embedding space gets diluted — a thousand-token chunk contains too many concepts for a single vector to represent faithfully.

Parent-child chunking is the pattern that dissolves this tradeoff.

The Architecture

Step 1 — Build parent chunks at structural boundaries

The first pass splits the document at natural structural seams: headings, sections, subsections. These become your parent chunks. They're large — potentially several thousand tokens — and represent a complete, coherent unit of thought in the document.

Parent chunks are not embedded. They live in a regular document store: MongoDB, SQLite, a JSON file on disk, or any key-value store. They're indexed by a parent ID.

Step 2 — Build child chunks from each parent

Each parent chunk is then split further into smaller, roughly equal-sized child chunks (~200–400 tokens each). Every child stores two things: its own text, and a parent_id reference pointing back to its parent.

Children are what you embed and index in your vector store (Milvus, ChromaDB, Pinecone, etc.).

Step 3 — Retrieve children, serve parents

At query time: run your semantic search against the child index. Collect the top-k child chunks. Look up their parent_id values. Fetch the corresponding parent chunks from your document store. Deduplicate. Pass the full parent chunks to the LLM.

The retriever got precision. The LLM gets context. Neither had to compromise.

💡 Key insight: retrieval and generation have genuinely different optimal chunk sizes. Parent-child chunking decouples those two concerns so you can optimise both independently.

Implementation

from langchain.text_splitter import RecursiveCharacterTextSplitter

    # 1. Parse document into section-level parent chunks
    parent_splitter = RecursiveCharacterTextSplitter(
        chunk_size=2000,
        chunk_overlap=100,
        separators=["\n## ", "\n### ", "\n\n", "\n"]
    )
    parent_chunks = parent_splitter.create_documents([full_text])

    # 2. Split each parent into small children
    child_splitter = RecursiveCharacterTextSplitter(
        chunk_size=300,
        chunk_overlap=30
    )

    child_chunks = []
    for i, parent in enumerate(parent_chunks):
        children = child_splitter.create_documents([parent.page_content])
        for child in children:
            child.metadata = {
                **parent.metadata,
                "parent_id": i,           # ← link to parent
                "parent_preview": parent.page_content[:120]
            }
            child_chunks.append(child)

    # 3. Embed and index child chunks only
    vector_store.add_documents(child_chunks)

    # 4. Store parent chunks separately
    for i, parent in enumerate(parent_chunks):
        doc_store.set(str(i), parent.page_content)

    # ── At retrieval time ──────────────────────────────────
    def retrieve_with_parent_context(query: str, top_k: int = 20):
        # Retrieve relevant child chunks
        child_hits = vector_store.similarity_search(query, k=top_k)

        # Collect unique parent IDs
        parent_ids = list({hit.metadata["parent_id"] for hit in child_hits})

        # Fetch full parent chunks
        context_docs = [doc_store.get(str(pid)) for pid in parent_ids]
        return context_docs

When to Use It — And When Not To

Use parent-child chunking when:

  • Your documents have rich structure — manuals, legal contracts, research papers, internal wikis
  • Tables or diagrams appear mid-section (a single row means nothing without the surrounding context)
  • Section titles carry semantic weight that shouldn't be lost during retrieval
  • Users ask questions that span multiple sentences or concepts within a section

Skip it when:

  • Documents are short and atomic — FAQ entries, product descriptions, support tickets
  • Every chunk is already a self-contained unit of meaning
  • You're latency-constrained and can't afford two storage lookups

Common Pitfalls

Forgetting to deduplicate

Multiple child chunks often share the same parent. If you retrieve five children all pointing to the same parent ID and blindly fetch the parent five times, you'll stuff the same 2,000-token section into your context window five times. Always deduplicate parent IDs before fetching.

Unbounded parent size

If your "structural boundary" splitting produces parents of 8,000+ tokens, you'll blow through your context window the moment you retrieve more than one or two. Cap parent chunk size — 1,500 to 3,000 tokens is usually a safe range — or use a sliding window approach where very large sections are split into overlapping parents.

Metadata orphaning

Child chunks need to inherit their parent's metadata: source document, page number, section title, creation date. If a child chunk only stores parent_id and nothing else, your citations and filtering capabilities break. Always propagate metadata down during the split step.

Skipping the re-ranker

Parent-child retrieval tends to pull back large chunks. Running a cross-encoder re-ranker on the child results before fetching parents helps you avoid pulling in irrelevant parent sections just because a tangentially-related child happened to score well.

Extending the Pattern

The parent-child idea extends naturally. You can build three-level hierarchies: document → section (parent) → paragraph (child). The document level stores author, date, and source. The section level stores the coherent argument. The paragraph level is what gets embedded. At retrieval time you might serve the section or even the full document depending on the query type.

This is essentially what systems like LlamaIndex's ParentDocumentRetriever implement under the hood — it's worth reading their source code once you've built a version yourself.
