Inverted Index vs BM25: When You Need Custom Keyword Search

BM25 doesn't understand logical AND/OR. Here's how an inverted index with a logical query parser gives you search that actually behaves.

Search ~ 8 min read

The gap: BM25 ranks documents by term frequency. It has no concept of AND, OR, or NOT. If your users need logical search — find all documents containing both X and Y — BM25 will disappoint them every time.

BM25 Is Excellent — At What It Does

BM25 (Best Match 25) is the backbone of almost every keyword search system built in the last two decades. Elasticsearch, OpenSearch, and Solr all use it as their default relevance scorer. It's battle-tested, fast, and surprisingly effective. For most search problems, it's the right choice.

BM25 works by scoring documents based on how frequently query terms appear in them, normalized by document length and calibrated by how rare those terms are across the corpus (IDF). Given the query "Panasonic maintenance manual", BM25 will return the documents where those terms appear most often — ranked by relevance.

Here's the problem: BM25 treats your entire query as a bag of words. It doesn't understand operators. It doesn't know that AND means both terms must appear. It has no concept of precedence, grouping, or logical constraints. Every word is just a signal contributing to a relevance score.

The Use Case That Broke BM25

Imagine a user who needs to find all documents that mention a specific aircraft part number and a specific vendor. Their natural language query might be:

"Find all documents containing Panasonic and part XYZ-4400"

What the user means: return only documents where both "Panasonic" AND "XYZ-4400" appear. Full stop.

What BM25 does: scores and ranks all documents containing any of the words — "find", "all", "documents", "containing", "Panasonic", "and", "part", "XYZ-4400". Documents mentioning only "Panasonic" score well. Documents mentioning only "part" also score. The word "and" is treated as just another term.

For a corpus of 14,000 technical documents, this produces noise. Lots of it.

The Inverted Index Solution

An inverted index flips the data model. Instead of organizing information as document → list of words, it organizes it as word → list of documents containing that word.

# Conceptually, an inverted index looks like this:
    index = {
        "panasonic": [doc_42, doc_117, doc_883, ...],
        "xyz-4400":  [doc_42, doc_201, doc_556, ...],
        "boeing":    [doc_11, doc_42, doc_77, ...],
        # ...
    }

When a user searches for "Panasonic", you do a direct dictionary lookup — O(1) — and get back the list of document IDs instantly. No scanning. No scoring loops over 14,000 documents.

The full data structure I built had three components:

  • documents — {index: chunk_text}, the actual text of each chunk
  • metadata — {index: {source, page, section, title, …}}, rich metadata per chunk
  • mapping — {word: [[doc_index, frequency], …]}, the inverted index itself

Building the Logical Query Parser

The inverted index solves the speed problem. The logical query parser solves the intent problem.

The input from users is natural language. The search engine needs a structured logical expression. A small LLM layer does this translation:

Input:  "List all documents containing Panasonic and part XYZ-4400"
    Output: "panasonic AND XYZ-4400"

    Input:  "Find docs about Boeing or Airbus maintenance with part ABC"
    Output: "(boeing OR airbus) AND ABC"

    Input:  "All contracts from 2023 not involving vendor XYZ"
    Output: "2023 AND NOT xyz"

The LLM was given a system prompt with clear rules and a bank of few-shot examples covering the common patterns. Output was always a structured logical string — no prose, no explanation.

That logical string then fed into a recursive descent parser that handled operator precedence and bracket grouping:

class LogicalQueryParser:
        """Parses expressions like: (A OR B) AND C AND NOT D"""

        def parse(self, query: str) -> set:
            tokens = self.tokenize(query.upper())
            return self.parse_or(tokens)

        def parse_or(self, tokens) -> set:
            left = self.parse_and(tokens)
            while tokens and tokens[0] == 'OR':
                tokens.pop(0)
                right = self.parse_and(tokens)
                left = left | right          # union
            return left

        def parse_and(self, tokens) -> set:
            left = self.parse_not(tokens)
            while tokens and tokens[0] == 'AND':
                tokens.pop(0)
                right = self.parse_not(tokens)
                left = left & right          # intersection
            return left

        def parse_not(self, tokens) -> set:
            if tokens and tokens[0] == 'NOT':
                tokens.pop(0)
                operand = self.parse_atom(tokens)
                all_ids = set(range(len(self.documents)))
                return all_ids - operand     # complement
            return self.parse_atom(tokens)

        def parse_atom(self, tokens) -> set:
            token = tokens.pop(0)
            if token == '(':
                result = self.parse_or(tokens)
                tokens.pop(0)  # consume ')'
                return result
            # Leaf node — look up in inverted index
            word = token.lower()
            if word not in self.mapping:
                return set()
            return {entry[0] for entry in self.mapping[word]}

When to Choose Each

The answer is usually both. They solve different problems:

  • Use BM25 / semantic search when users want relevant results ranked by how well they match a natural language question
  • Use an inverted index with logical parsing when users need to know "which documents contain this specific combination of terms" — compliance checks, part number lookups, contract searches

In many real systems, you expose both as separate search modes and let users pick based on their intent. That's exactly what we did — a toggle on the UI between "Ask a question" (RAG) and "Keyword search" (inverted index).

A Note on Scaling

This approach works beautifully up to a few hundred thousand documents. Beyond that, your in-memory inverted index starts consuming significant RAM, and you'll want to move to a proper search engine like Elasticsearch or Typesense that implements inverted indexes at scale with disk-backed storage and distributed sharding.

But for enterprise document search at the 10K–200K document range, a well-built Python inverted index is fast, transparent, and requires zero infrastructure dependencies.
